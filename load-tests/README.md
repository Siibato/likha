# Likha Load Tests

k6 load testing suite for the Likha LMS server. TypeScript + pnpm + esbuild.

## Prerequisites

```bash
brew install k6
```

## Setup

```bash
# 1. Install dependencies
pnpm install

# 2. Configure environment
cp .env.example .env
# Edit .env — set BASE_URL to your Pi's IP address

# 3. Generate seed manifest (resets DB, seeds, and exports manifest in one step)
# From the project root:
make db-seed-manifest

# 4. Build all scenarios
pnpm build
```

## Running Tests

```bash
# Increase socket limit (macOS — required before each k6 run)
ulimit -n 65535

# Baseline: verify server is healthy (50 VU, 2 min)
pnpm test:baseline

# Stress: find breaking point (50 → 500 VU ramp)
pnpm test:stress

# Classroom mix: teachers + students concurrent (75 VU)
pnpm test:classroom

# Exam surge: 70 students burst into same assessment simultaneously
pnpm test:exam

# Delta sync: delta vs full sync comparison (100 VU)
pnpm test:delta

# Teacher grading: grading burst + grade compute (5 VU)
pnpm test:grading

# Quick profiler: all endpoints, 50 VU, 1 min — fast outlier detection
pnpm test:all-endpoints-quick

# Run all in sequence
pnpm test:all

# Export results to CSV
export $(grep -v '^#' .env | xargs) && k6 run --out csv=results.csv dist/stress.js
```

## Seed Manifest

`seed-manifest.json` is generated by the server and contains all seeded entity IDs
(classes, assessments, assignments, users). It is gitignored — regenerate it whenever
you reset the database.

```bash
# Reset DB + seed + export manifest in one go (from project root)
make db-seed-manifest
```

## Realistic 150-User Load Test Specification

The realistic-150 scenario simulates steady-state classroom traffic against the Likha LMS server running on a Raspberry Pi, ramping from 0 to 150 concurrent virtual users (VUs) over a 7-minute window. The ramp profile progresses in four stages: 30 seconds to 50 VUs, 1 minute to 100 VUs, 3 minutes to the full 150 VUs, a 2-minute hold at peak, and a 30-second ramp-down to zero. This gradual ramp allows the server to warm up connections and caches before sustaining peak load, mirroring how a real school day unfolds as teachers and students log in sequentially rather than all at once. The test is executed with k6 against the server's REST API over the local network, with the macOS file descriptor limit raised to 65,535 (`ulimit -n 65535`) to avoid socket exhaustion at peak concurrency. Prior to each run, the database is reset and reseeded via `make db-seed-manifest`, which populates 5 teachers, 70 students, and 1 admin account, along with all associated classes, assessments, assignments, and grading data. The generated `seed-manifest.json` provides the scenario with real entity IDs so that every request targets valid resources. During the k6 setup phase, all 150 VUs are authenticated upfront—7 admins, 38 teachers, and 105 students—yielding a role distribution of approximately 70% students, 25% teachers, and 5% admin, consistent with a real campus where students outnumber staff. Each VU performs an initial full sync on its first iteration to simulate a fresh device login, then enters a steady-state loop with 2–5 seconds of think time between actions. The traffic mix is weighted to reflect natural usage: 85% of iterations issue GET reads (navigating classes, assessments, assignments, grades, and materials), 10% perform delta syncs (app restart or reconnect), 4% push data to the server (grade scores from teachers, submission uploads from students), and 1% trigger a full sync (rare re-login or data expiry). Endpoint-specific thresholds enforce p(95) latency targets ranging from 200ms for lightweight metadata calls to 3000ms for full sync, with an overall error rate ceiling of 5% and a check pass rate above 90%. It should be noted that this test measures server-side response latency only—the time from request received to response sent—and does not account for client-side persistence overhead such as SQLite writes or local cache hydration on the mobile device, which would add additional latency in a real deployment.

Under peak load at 150 concurrent users, the server demonstrates strong performance for read-heavy operations. GET endpoints and sync push operations exhibit average latencies below 30ms, indicating that the Rust backend efficiently handles concurrent reads and write-through syncs under sustained load. Delta sync, which computes incremental changes since a client's last sync timestamp, averages approximately 430ms—a noticeable but acceptable delta given the computational cost of diffing server state across multiple entity tables. Full sync, which aggregates and serializes all class data for a device, averages around 712ms, reflecting the heavier payload assembly and database queries involved in bundling an entire dataset. The latency gap between delta and full sync (roughly 280ms) validates the architectural decision to favor delta sync for routine updates and reserve full sync for initial login or recovery scenarios. No 5xx errors were observed during the steady-state hold, and all endpoint-specific p(95) thresholds were satisfied, confirming that the Raspberry Pi can sustain realistic classroom concurrency without degradation.
